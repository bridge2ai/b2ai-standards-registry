Topic Hierarchy Widget — visual mock

Visual mock of the topic hierarchy widget for the b2ai.standards portal's topic details pages. Renders the DataTopic DAG with the user's currently-chosen topic highlighted, plus the ancestry and descendants in its neighborhood. Earlier iterations (Parts A/B/B′/C) are kept as collapsed history below.

How it behaves

  1. The chosen topic is the one you're viewing. It's highlighted in amber. Clicking any other topic name navigates to that topic's page — making it the new chosen topic. Switching topics is like loading a new page: any expand/collapse state in the widget resets.
  2. The default view shows just enough. When you land on a topic, the widget shows the chosen topic plus its immediate parents and immediate children. Everything else is hidden until you ask for it.
  3. Reveal more by expanding. Each row has a ▶ N toggle showing how many children it has. Click to show them. Click ▼ N to hide them again.
  4. Hidden ancestors show as a breadcrumb. If a row's parent isn't shown in the widget, the path above it appears as a clickable trail like Data / Biology /. Click any segment to bring that ancestor into the widget as a row.
  5. The chosen topic stays visible. Anything else can be collapsed away, but you can't hide the row you're on. If collapsing an ancestor would hide it, the toggle drops to ◐ N instead — meaning "only the child on the path to the chosen topic is showing."
  6. Topics with multiple parents appear once. A few topics (3 of 52 in the b2ai data) have more than one parent. To avoid duplication, each renders under one canonical parent, with a "★ also under: other-parent" note and a visual rail connecting it to its other parent(s) on screen.
How it works under the hood (implementation detail; not needed for design feedback)
  1. Each row sits at its real DAG depth. The longest path from root determines its indent level. The └ glyph in front of each row makes the parent-child edge explicit, in case the indentation alone isn't enough of a cue.
  2. Path-connectivity invariant. If a row and one of its ancestors are both visible, every intermediate ancestor between them is also visible. This is what makes ancestor rows fall back to ◐ instead of disappearing when collapsed.
  3. Canonical parent = longest path to root. When a topic has multiple parents, it's rendered under the parent whose path to root is longest. Shorter-path parents get the skip-rail. This keeps the visual tree a tree, with the polyhierarchy edges drawn as annotations on top.
  4. Default ordering: by topic ID, with one override — for the chosen topic's parents specifically, the shorter-path parent is rendered above the longer-path one. Without this, polyhierarchy topics would show their longer-path parent above the shorter-path one and confuse the user. A user setting can switch to pure alphabetical.

Earlier explorations — nested colored boxes (Parts A, B, B′)

What we tried: Render each row's parent or child expansion as a colored adjacent box, nested inside the row's containing context. The anchor T sat inside its own outermost box (darkest tint), with its ↑ expansion as a lighter box above and ↓ expansion as a lighter box below. Nested expansions (a child's children, etc.) produced further-lighter boxes recursively.

What was good about it: The boxes made each expansion's scope visually unambiguous — "this region of the page is X's expansion." Colored depth gradient gave the user a strong "outer = more important" cue. Toggle pills with embedded counts (▲ N, ▼ N) were intuitive. Several details from this iteration carried forward to Part D — the counts-in-toggles, the highlighted-anchor styling, the secondary-parent ★ notation, breadcrumb-like path-from-root, the duplicate-row indicator for polyhierarchy.

Why we moved on: Polyhierarchy mid-tree (B6 case: when a child has another parent whose ↑ would be expanded) required visually splitting a parent box around the inner expansion. Two halves of the same conceptual box with a different-colored block squeezed between them. That problem doesn't have a clean visual solution within the box-nesting idiom. The boxless approach in Part D sidesteps it entirely by representing parent-child edges with rails/glyphs instead of containment.

Things worth revisiting if we return to box-style: the four color schemes (stacked alpha, monotonic gradient, warm gradient, cycling palette) all have their merits; the toggle-with-counts pattern; the secondary-parent note style.

Part B′: Deeper nesting examples

Goal: probe the box-based widget on cases it hadn't handled yet. B5 fabricated a topic with both multiple parents AND multiple children (real data has none). B6 attempted grandchild ↓ expansion + mid-sibling ↑ expansion — and surfaced the structural problem: Metabolome's ↑ expansion (revealing its other parent) needed to render at indent 1 (same as Omics), which forced visually splitting the anchor's ↓ box around it. This is what killed the box approach.

B5 is fabricated (a polyhierarchical topic with children) — the real dataset has only leaf topics with multiple parents, so we synthesize one to exercise the harder case. B6 shows what happens when a child's child is expanded — recursive nesting of ↓ boxes with depth incrementing.
